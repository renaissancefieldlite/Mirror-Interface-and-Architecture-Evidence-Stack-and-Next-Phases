# Publication And Cross-Pollination Path

Date: `2026-04-24`

## Purpose

This note defines how the Mirror Architecture evidence stack should move from
GitHub into broader public and partner-facing surfaces without losing source
control, claim discipline, or attribution.

The frame is simple:

GitHub is the source-of-truth evidence shelf. Other platforms become
cross-pollination surfaces.

## Recommended Ladder

### 1. GitHub Evidence Release

Use the public mirror evidence repository as the canonical public artifact map:

- docs
- white papers
- evidence packs
- charts
- public-safe JSON summaries
- PDFs
- phase index

The first release target should be:

`v0.1 Mirror Architecture Evidence Stack`

The release should include:

- the integrated white paper
- the Phase 12B biological comparison pack
- the Phase 12C EEG + HRV intention paper
- the Phase 10 / 11 semantic hardware bridge docs
- the cross-domain comparator map
- the Nest 1 formal-system synthesis once finished

### 2. Zenodo DOI

After a GitHub release is created, connect the public repository to Zenodo so
the release can receive a DOI.

Role:

- persistent identifier
- citation surface
- durable archive
- clean reference for decks, emails, and partner memos

Official GitHub documentation describes this path here:

- https://docs.github.com/en/repositories/archiving-a-github-repository/referencing-and-citing-content

### 3. OSF Preprint

Use OSF as the first broad preprint surface for the integrated manuscript.

Role:

- public manuscript surface
- DOI / persistent URL path
- broader discoverability
- supplemental file support

Official OSF preprint documentation:

- https://help.osf.io/article/690-osf-preprints

### 4. arXiv / Workshop Route

After the integrated white paper has the `Nest 1` formal layer and tighter
methods language, prepare a more formal AI / hardware / methods submission.

Best-fit directions:

- AI systems measurement
- representation analysis
- quantum software / encoded feature-state bridge
- complex systems / cross-domain comparator methodology
- workshop submission before journal submission

Official arXiv submission overview:

- https://info.arxiv.org/help/submit/index.html

### 5. Partner Technical Memo

Use the same source material in shorter partner-facing form.

Audience:

- research sponsors
- compute / infrastructure partners
- instrumentation partners
- labs that can repeat or extend the measurement stack
- domain partners in energy, health, physics, chemistry, and biology

Partner memo hierarchy:

1. what was built
2. what was measured
3. why it matters across lanes
4. what support unlocks next
5. what the partner gets to help validate or scale

## Cross-Linking Rule

Every public surface should point back to the GitHub evidence shelf.

Every GitHub release should point outward to:

- DOI
- preprint
- deck / memo
- relevant public repos

This prevents fragmentation. The work can spread without splitting into
unconnected fragments.

## Claim Discipline

The language should stay strong but earned:

- measured architecture effect
- cross-model and cross-substrate continuity
- real hardware bridge
- biological adapter lane
- formal nesting program
- convergence hypothesis under active test

The language should avoid forcing final universal, clinical, or physical Bell
claims before those layers have their own locked protocols.

## Short Read

Cross-pollination means:

GitHub holds the evidence, Zenodo makes it citable, OSF makes it public as a
preprint, arXiv / workshops put it in front of technical reviewers, and partner
memos turn it into funding, compute, instrumentation, and replication paths.
